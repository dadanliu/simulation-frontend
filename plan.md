# Next.js 线性全链路时间线计划

## 1. 目标

这份文档只回答一个问题：

> 人在 VSCode 里写下前端源码之后，到用户看到页面、再到用户点击后页面响应，中间到底按什么顺序发生了什么。

所以这里不再按 `RSC / SSR / SSG / CSR / Hydration` 这些术语分章。

这里的唯一组织方式是：**严格按真实发生顺序展开。**

---

## 2. 总链路

```text
人在 VSCode 里写前端源码
-> 源码描述状态 / 结构 / 样式 / 交互
-> 构建工具读取源码并分析依赖
-> TypeScript / JSX / 模板语法被编译成可执行 JavaScript
-> CSS / 图片 / 字体等资源被整理、转换、切分
-> 产出 HTML / JS / CSS / 静态资源 / 服务端运行逻辑
-> dev server / Node / Nginx / CDN 把这些文件暴露给浏览器
-> 浏览器发起请求并收到字节流
-> 浏览器解析 HTML，生成 DOM
-> 浏览器解析 CSS，生成 CSSOM
-> 浏览器加载并执行 JavaScript
-> 框架运行时启动，创建应用实例
-> 框架读取状态 / 组件树 / 路由 / 数据
-> 计算当前状态下页面应该长什么样
-> 生成或更新 DOM
-> 浏览器把 DOM + CSSOM 组合成 Render Tree
-> 浏览器进行 Style 计算
-> 浏览器进行 Layout
-> 浏览器进行 Paint
-> 浏览器进行 Composite
-> GPU / 显示系统输出到屏幕
-> 用户看到页面
-> 用户点击 / 输入 / 滚动
-> 浏览器把输入转换成事件
-> JavaScript 事件处理函数执行
-> 应用状态变化
-> 框架重新计算受影响的 UI
-> 更新 DOM / 样式 / 布局 / 绘制
-> 屏幕像素变化
-> 用户看到系统响应
```

---

## 3. 六个连续阶段

## Stage 1：源码描述阶段

### 位置
整条链路的起点。

### 输入
- 页面需求
- 状态设计
- 结构设计
- 样式设计
- 交互设计

### 输出
- React / Next.js 源码
- TypeScript 类型
- 组件树
- 路由定义
- CSS 与静态资源引用

### 本质
源码不是页面本身，而是“页面如何生成、如何更新”的规则描述。

### 应拆场景
- 源码如何描述结构
- 源码如何描述状态
- 源码如何描述样式
- 源码如何描述交互

---

## Stage 2：构建处理阶段

### 位置
源码完成之后、浏览器请求之前。

### 输入
- `app/`
- `components/`
- `lib/`
- `package.json`
- CSS / 图片 / 字体等资源

### 输出
- 编译后的客户端 JS
- 服务端运行逻辑
- 拆分后的 CSS
- HTML / 预渲染结果
- 构建产物清单

### 本质
构建工具把“开发时语义”转换成“运行时可以交付的产物”。

### 应拆场景
- 依赖图分析
- TS / JSX 编译
- 静态资源整理
- 构建产物输出

---

## Stage 3：资源交付与网络请求阶段

### 位置
构建完成之后，浏览器开始拿资源时。

### 输入
- HTML
- JS
- CSS
- 图片
- 字体
- 框架协议数据或接口响应

### 输出
- 浏览器收到的响应字节流

### 本质
服务器交付的是字节，不是“组件概念”。

### 应拆场景
- dev server 如何暴露资源
- 浏览器请求后拿到的是哪些字节

---

## Stage 4：浏览器解析与首屏渲染阶段

### 位置
浏览器拿到首批资源之后，到用户首次看到页面之前。

### 输入
- HTML 字节
- CSS 字节
- JS 字节

### 输出
- DOM
- CSSOM
- Render Tree
- 首屏像素

### 本质
页面显示出来，是浏览器渲染流水线的结果，不是框架直接“把像素画到屏幕上”。

### 应拆场景
- HTML -> DOM
- CSS -> CSSOM
- JS 执行与运行时启动
- Render Tree / Style / Layout / Paint / Composite

---

## Stage 5：框架接管与可交互阶段

### 位置
首屏出现之后，到页面真正可交互时。

### 输入
- 首屏 HTML
- 客户端 JS
- 框架运行时

### 输出
- 已绑定事件的可交互页面

### 本质
“能看到页面”和“页面能点”是两个不同阶段。hydration 就在这里。

### 应拆场景
- hydration 的时间位置
- 框架运行时接管页面
- 首屏可见但未接管 vs 已接管可交互

---

## Stage 6：用户交互与增量更新阶段

### 位置
页面已经可交互之后。

### 输入
- 用户点击 / 输入 / 滚动
- 当前应用状态
- 必要时的接口数据

### 输出
- 更新后的状态
- 更新后的 DOM
- 新一轮样式 / 布局 / 绘制结果
- 更新后的屏幕像素

### 本质
用户交互触发的是一条局部更新链，而不是把全站从头再跑一遍。

### 应拆场景
- 点击触发事件
- 本地状态更新
- DOM 更新
- 浏览器重新 style / layout / paint / composite
- `fetch -> Route Handler -> JSON -> UI 更新`

---

## 4. 主项目应该怎么承载这条链

`full-chain/` 只负责讲一条完整故事，不再做概念拼盘。

首页应该按下面顺序组织：

1. 源码描述区
2. 构建处理区
3. 产物与交付区
4. 浏览器解析区
5. 首屏渲染区
6. hydration / 可交互区
7. 用户交互更新区
8. 请求往返更新区

要求：

- 每一段都说明它在主链中的位置
- 每一段都写清输入和输出
- 页面上能看到真实状态变化或时间戳变化

---

## 5. 独立场景应该怎么承载这条链

`scenarios/` 的职责是把主链上的单点节点拆开。

每个场景都必须明确：

1. 它在线性链路中的位置
2. 它的输入是什么
3. 它的输出是什么
4. 它模拟的是哪个真实节点
5. 它和前后节点如何衔接

如果一个场景做不到这 5 点，它就还不是一个清楚的链路场景。

---

## 6. 术语在新方案中的位置

以下术语仍然保留，但降级为“局部解释”，不再担任最高层目录：

- `RSC`
- `SSR`
- `SSG`
- `CSR`
- `Hydration`
- `Route Handler`

新的原则是：

- 主线负责回答“先发生什么，后发生什么”
- 术语负责回答“这一小段为什么会这样工作”

---

## 7. 验收标准

如果这份计划被正确落地，读者看完整个项目后应该能顺着说出下面这 7 句：

1. 我在编辑器里写下的是源码规则，不是页面本身。
2. 构建把源码变成了浏览器和服务器可消费的产物。
3. 服务器交给浏览器的是 HTML / JS / CSS / 其他资源字节。
4. 浏览器把字节解析成 DOM / CSSOM，并执行 JS 启动运行时。
5. 浏览器通过 Render Tree / Style / Layout / Paint / Composite 画出首屏。
6. hydration 完成后，页面才真正进入可交互状态。
7. 用户交互会触发事件、状态变化、DOM 更新和后续渲染流水线。

只要项目还不能支持读者完整复述这 7 句，就说明文档和代码还没有真正按线性主链组织好。
