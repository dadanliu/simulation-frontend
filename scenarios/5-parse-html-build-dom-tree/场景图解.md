# 场景 5 图解：浏览器解析 HTML，生成 DOM 树

---

## 一、整体结构

这一跳只有一个进程内部的流水线，输入是一段 HTML 字符流，输出是一棵 DOM 树。

```
demo-input/index.html  (HTML 字节流，UTF-8)
        │
        ▼
┌──────────────────┐
│  ① Tokenizer     │   按有限状态机读字符，切成 token
│  字符流 → token  │   （StartTag / EndTag / Text / Comment / Doctype）
└──────────────────┘
        │
        ▼
┌──────────────────┐
│  ② Tree          │   维护一个「开元素栈」，
│  Constructor     │   按 token 把节点串成父子关系
│  token → DOM     │
└──────────────────┘
        │
        ▼
   Document 节点树（#document → <html> → <head>/<body> → …）
        │
        └──►  下一跳：CSSOM 构建 / 脚本执行 / 样式计算 / 布局 / 绘制
```

> 本场景只负责把 "HTML 字符串" 变成 "DOM 对象树"，
> 不负责 CSS 解析、样式计算、JS 执行、布局和绘制。
>
> 如果你想先解决“为什么浏览器不等整棵树建完再继续”的疑问，
> 请先看 [`为什么先-tokenize-再-build-dom-图解.md`](./为什么先-tokenize-再-build-dom-图解.md)。

## 二、Tokenizer 的状态机

Tokenizer 本质上就是一个"一次看一个字符、按当前状态决定下一步"的循环。
下面是手写版里实际用到的几种状态（和 WHATWG HTML 规范里的 Tokenization 章节同构）。

> **想看 15 个状态一一展开、逐字符动作表、和 WHATWG 规范的完整对照**，
> 请看同目录的 [`tokenizer-状态机详解.md`](./tokenizer-状态机详解.md)，本节只给主干骨架。

```
                        ┌──────────┐
        其他字符累积     │          │
        ┌──────────────►│   DATA   │◄──────────────────┐
        │               └─────┬────┘                   │
        │                     │ '<'                    │ '>'
        │                     ▼                        │
        │               ┌──────────┐                   │
        │               │ TAG_OPEN │                   │
        │               └─────┬────┘                   │
        │       '/'     ┌──────┴──────┐  '!'           │
        │       ┌───────┤            ├────────┐        │
        │       ▼       │ 字母        │        ▼        │
        │  ┌──────────┐ ▼             │ ┌─────────────┐ │
        │  │END_TAG_  │┌──────────┐   │ │ MARKUP_DECL │ │
        │  │  OPEN    ││ TAG_NAME │   │ │  <!-- 或    │ │
        │  └────┬─────┘└────┬─────┘   │ │  <!doctype  │ │
        │       │ 字母      │ 字母     │ └─────┬───────┘ │
        │       ▼           │          │       ▼         │
        │  ┌──────────┐     │空白      │ ┌────────────┐ │
        │  │ TAG_NAME │     ▼          │ │  COMMENT   │─┘
        │  └────┬─────┘┌──────────────┐│ │  DOCTYPE   │
        │       │空白/>│BEFORE_ATTR_  ││ └────────────┘
        │       │      │   NAME       ││
        │       │      └──┬───────────┘│
        │       │         │字母         │
        │       │         ▼            │
        │       │   ┌──────────┐       │
        │       │   │ATTR_NAME │       │
        │       │   └─┬────────┘       │
        │       │     │'='             │
        │       │     ▼                │
        │       │ ┌────────────────┐   │
        │       │ │BEFORE_ATTR_VAL │   │
        │       │ └──┬─────────────┘   │
        │       │    │                  │
        │       │    ├── '"' → ATTR_VALUE_DQ  ──┐
        │       │    ├── "'" → ATTR_VALUE_SQ  ──┤
        │       │    └── 其他 → ATTR_VALUE_UNQ─┤
        │       │                               ▼
        │       │                       值结束 → BEFORE_ATTR_NAME
        │       ▼
        │  遇 '>' 发 StartTag/EndTag，回 DATA  ───────────┘
        │
        └──── 整段文本累积在 textBuf，遇下一个 '<' 时 emitText()
```

关键点：

- 「遇到某个字符时做什么」完全由**当前状态**决定，这就是"有限状态机"的定义。
- `emitXxx()` 代表一个 token 构造完成，被放进 `tokens[]` 输出给下游。
- 实体解码（`&` / `☃`）、Raw Text（`<script>` 内部不当作标签）在真实浏览器里是独立状态分支，手写版省了。

## *三、Tokenizer 切词示例*

*对一段* `<p>Hello <em>DOM</em></p>`*，tokenizer 会产出：*

```
字符流：  < p > H e l l o _ < e m > D O M < / e m > < / p >

状态轨迹：DATA → TAG_OPEN → TAG_NAME → DATA → TAG_OPEN → TAG_NAME → DATA
         → TAG_OPEN → END_TAG_OPEN → TAG_NAME → DATA
         → TAG_OPEN → END_TAG_OPEN → TAG_NAME → DATA

Token 流：
  [0] StartTag <p>
  [1] Text     "Hello "
  [2] StartTag <em>
  [3] Text     "DOM"
  [4] EndTag   </em>
  [5] EndTag   </p>
```

## 四、Tree Constructor 的「开元素栈」

有了 token 流后，构建树的核心是一个**只做两件事的栈**：

```
规则 1：遇到 StartTag
   · 创建一个 Element 节点
   · 把它 append 到「栈顶节点」的 children
   · 如果是 void 元素（img/meta/link/br/hr/input…）或自闭合，不压栈
   · 否则压栈（它变成新的栈顶 = 新的「当前插入点」）

规则 2：遇到 EndTag
   · 从栈顶往下找同名 Element
   · 找到就把它及其以上的全部出栈

规则 3：遇到 Text / Comment
   · 直接 append 到栈顶节点的 children

规则 4：遇到 Doctype
   · 直接 append 到 Document
```

栈在整个解析过程中就是「当前的祖先链」，栈顶就是「现在要插在哪个元素下面」。

## 六、void 元素为什么不压栈？

HTML5 规范定义了这几类**不可能有子节点**的元素，叫 void elements：

```
area  base  br  col  embed  hr  img  input
link  meta  param  source  track  wbr
```

如果也把它们压栈，后续任何节点都会被插进它们里面，违反语义。
所以规则非常干脆：**只挂成子节点，不改变栈**。

手写版里的表现：

```js
const VOID_ELEMENTS = new Set(['area','base','br','col','embed','hr','img','input',
                               'link','meta','param','source','track','wbr']);

if (!VOID_ELEMENTS.has(tagName) && !selfClosing) {
  stack.push(el);
}
```

## 七、错乱 HTML 时会发生什么？

真实网页里经常出现：

```html
<p><b>粗体
<p>新段落
```

浏览器（和 parse5）会按"adoption agency algorithm"把 `<b>` 自动续到下一段：

```
<p>
  <b>粗体</b>
</p>
<p>
  <b>新段落</b>
</p>
```

手写版**不会**做这件事——它只会老老实实按栈匹配。
这就是"流行方案比手写版多做的那一大块工程能力"的典型例子，
手写版负责让你**看清正常情况下发生了什么**，
流行方案负责**把所有异常也都处理到位**。

## 八、这一跳的边界

```
上一跳（场景 4）：字节流到达浏览器
   └─ Buffer：<!doctype html><html>…

本跳：
   · 输入 = 字符串（解码过的 HTML 文本）
   · 输出 = DOM 树（Document / Element / Text / Comment 节点）
   · 不负责：解析 CSS、下载/执行 JS、布局、绘制

下一跳：
   · 解析 CSS → CSSOM
   · 合并 DOM + CSSOM → Render Tree
   · 布局 → 绘制 → 合成
```

## 九、一句话总结

```
HTML 解析 = 有限状态机（切词） + 一个栈（串成树）
                 │                      │
                 └── Tokenizer          └── Tree Constructor

字符流 ──► Token 流 ──► 开元素栈推/出 ──► DOM 树
```

手写版把这两步都写得极薄，`debug/step-by-step.js` 允许你把栈的每一次 push/pop 都肉眼看到；
parse5 在同一条主干上又覆盖了插入模式、错误恢复、命名空间切换等全部边界分支。
