# 缓存策略、ETag、304、nginx 与 CDN 说明

本文整理这个场景里几个容易混在一起的概念：

- 浏览器缓存策略
- `ETag` / `Last-Modified` / `If-Modified-Since`
- 为什么会返回 `304 Not Modified`
- nginx 的作用
- CDN 的作用
- 它们在一条请求链路里的关系

---

## 1. 先说结论

这个场景里的核心是：

- 浏览器本地**可以缓存资源**
- 但不是每次都直接用
- 浏览器会带上条件头去问服务器：**“我这份缓存还能继续用吗？”**
- 如果资源没变，服务器就返回 **`304 Not Modified`**
- 浏览器继续使用本地缓存副本，不重新下载响应体

这类机制叫：

- **协商缓存**
- 或 **缓存再验证（revalidation）**

---

## 2. 什么是缓存策略

HTTP 缓存大体可以分成两类：

### 2.1 强缓存

典型头：

```http
Cache-Control: max-age=3600
```

含义：

- 在 3600 秒内，浏览器直接使用本地缓存
- 连请求都不发给服务器

表现：

- DevTools 里可能看到 `from disk cache` 或 `from memory cache`
- 不会看到 304，因为请求可能根本没发出去

---

### 2.2 协商缓存

典型组合：

```http
Cache-Control: no-cache
ETag: "..."
Last-Modified: ...
```

含义：

- 浏览器可以保存本地副本
- 但每次使用前都要先去服务器确认这份副本是否还有效

表现：

- 浏览器发请求
- 带上 `If-None-Match` / `If-Modified-Since`
- 服务器判断资源没变时，返回 `304 Not Modified`

---

## 3. `Cache-Control: no-cache` 到底是什么意思

这个头最容易被误解。

很多人会把 `no-cache` 理解成：

- “不要缓存”

但实际上更准确的意思是：

- **可以缓存**
- **但是使用前必须先向服务器重新验证**

所以 `no-cache` 并不等于“完全不缓存”。

它更像：

> 缓存可以留着，但别擅自用；每次先问我。

---

## 4. 为什么会返回 `304 Not Modified`

当浏览器已经缓存过某个资源，下次再请求时，可能会带上这样的头：

```http
If-None-Match: "69df0251-286"
If-Modified-Since: Wed, 15 Apr 2026 03:13:21 GMT
```

它的意思分别是：

### 4.1 `If-None-Match`

意思是：

> 如果资源当前的 `ETag` 还是这个值，说明没变；那你就别再发正文给我了。

---

### 4.2 `If-Modified-Since`

意思是：

> 如果这个时间之后资源没改过，那你也别再发正文给我了。

---

### 4.3 服务器如何决定 304

服务器会拿当前资源的版本信息和请求头里的条件做比较：

- 当前 `ETag` 是否仍然匹配
- 当前 `Last-Modified` 是否仍然没有晚于请求给出的时间

如果判断资源没变，就返回：

```http
304 Not Modified
```

含义是：

- 资源没有变化
- 浏览器继续使用本地缓存
- 响应通常不带 body

所以 **304 不是错误**，而是“缓存可复用”的成功判断结果。

---

## 5. `ETag` 是什么

`ETag` 可以理解成：

- **资源版本号**
- **资源当前状态的标识符**

例如：

```http
ETag: "69df0251-286"
```

浏览器会在下次请求时带上：

```http
If-None-Match: "69df0251-286"
```

服务器再判断：

- 如果当前资源的 ETag 还是这个值 -> `304`
- 如果已经变了 -> `200` + 新内容 + 新 ETag

---

## 6. `ETag` 由什么决定

协议只规定：

- `ETag` 是资源版本标识

但**具体怎么生成，由服务器自己决定**。

常见生成方式有三类：

### 6.1 内容哈希

比如对文件内容做：

- MD5
- SHA1
- SHA256

特点：

- 内容只要变一点，ETag 就会变
- 版本表达比较准确

---

### 6.2 文件元数据组合

比如用：

- 文件最后修改时间（mtime）
- 文件大小
- inode 或其他文件标识

特点：

- 生成开销小
- 静态文件服务里很常见

---

### 6.3 业务层版本号

动态接口里常见，比如：

- 数据版本号
- 数据库行版本
- 发布版本号
- Git commit hash

---

## 7. 这个场景里的 ETag 大概率是什么

像这样的值：

```http
ETag: "69df0251-286"
```

看起来很像 nginx 风格静态资源的 ETag 表达。

通常不是“明显的内容哈希串”，更可能接近：

- 文件修改时间
- 文件大小

这类组合生成的版本标识。

如果响应头里还有类似：

- `Server: nginx/1.28.3`
- `X-Delivery-Mode: nginx-style-static-server`

那么基本可以认为它是：

- **nginx 或 nginx 风格静态服务器**
- 基于静态文件元数据在做缓存校验

注意：

- 这是**大概率判断**
- HTTP 协议本身并不强制 ETag 的具体算法

---

## 8. `Last-Modified` 是什么

`Last-Modified` 表示：

- 资源最后修改时间

例如：

```http
Last-Modified: Wed, 15 Apr 2026 03:13:21 GMT
```

浏览器会把它带回去：

```http
If-Modified-Since: Wed, 15 Apr 2026 03:13:21 GMT
```

服务器据此判断：

- 这个时间之后有没有改动

---

## 9. 为什么 `ETag` 和 `Last-Modified` 常常一起出现

理论上：

- **只用 ETag 也可以完成缓存校验**

但实际工程里经常两个一起发，原因主要有四个。

### 9.1 两种信息维度不同

- `ETag`：版本维度
- `Last-Modified`：时间维度

它们很像，但不完全重复。

---

### 9.2 兼容性更好

有些客户端、代理、缓存层，对时间校验支持更自然；有些对 ETag 处理更好。

两个都给，相当于：

- 双保险
- 提高兼容性

---

### 9.3 有降级能力

如果某次请求没带 `If-None-Match`，还可以看 `If-Modified-Since`。

反过来，如果时间判断不可靠，也还能靠 ETag。

---

### 9.4 `Last-Modified` 成本低且可读

读取静态文件最后修改时间几乎没有额外成本。

而且它对调试很有帮助：

- 人能直接看懂文件最后更新时间
- 比一串 ETag 更直观

---

## 10. 两者哪个更强

通常可以这样理解：

- **ETag 更精确**
- **Last-Modified 更粗粒度**

### 10.1 `Last-Modified` 的局限

常见问题：

- 通常只有秒级精度
- 快速连续修改可能区分不出来
- 有时文件时间变了但内容没变
- 有时内容变了但时间戳处理不规范

---

### 10.2 `ETag` 的优势

- 更像真正的版本号
- 不一定依赖时间精度
- 对动态资源也适用
- 更适合表达“是否同一版”

因此现实里通常会把 `ETag` 当成更主要的校验依据，而 `Last-Modified` 作为补充和兜底。

---

## 11. 既然 ETag 更强，为什么不只留 ETag

因为工程实践追求的往往不是“最少机制”，而是“最稳机制”。

只留 ETag 当然可以，但会有几个问题：

- 全部正确性都押在 ETag 生成逻辑上
- 某些客户端或中间层对 ETag 支持未必理想
- 少了一个低成本、通用、可读的辅助信号

所以很多服务器的默认选择是：

- **ETag 负责精确判断**
- **Last-Modified 负责补充和兼容**

---

## 12. 服务器同时收到两个条件头时怎么判

通常情况下：

- **优先看 `If-None-Match`（ETag）**
- `If-Modified-Since` 作为补充或后备

也就是：

1. 浏览器带上 `If-None-Match`
2. 服务器先比较当前 ETag 是否匹配
3. 匹配 -> 可以返回 304
4. 不匹配 -> 返回 200 和新内容

这意味着：

- 在“两者都在”的情况下，**ETag 通常是主判断依据**
- 时间头更多是辅助信息

---

## 13. 为什么首页 HTML 常用协商缓存

像 `/` 或 `index.html` 这种 HTML 入口文件，通常不适合长期强缓存。

原因是：

- HTML 经常会引用 JS/CSS 资源
- 发布新版本时，HTML 往往是入口
- 如果 HTML 被强缓存太久，用户可能拿到旧入口页面

所以 HTML 常见策略是：

- 允许本地缓存
- 但每次访问都重新验证
- 没变就 304
- 变了就 200 新内容

也就是：

- **HTML 常用协商缓存**

---

## 14. 为什么 JS/CSS 更常见“强缓存 + 指纹文件名”

对这类静态资源，常见做法是：

- 文件名带 hash
- 例如 `app.abc123.js`

然后配合：

```http
Cache-Control: max-age=31536000, immutable
```

原因是：

- 文件名一变，就代表内容版本变了
- 老文件可以长时间强缓存
- 新版本通过新文件名来获取

这样可以做到：

- 浏览器几乎不必重复验证
- 加载效率高
- 版本切换明确

所以常见搭配是：

- **HTML：协商缓存**
- **带 hash 的 JS/CSS/图片：强缓存**

---

## 15. nginx 的作用是什么

nginx 通常有四种核心角色。

### 15.1 Web 服务器

直接提供 HTTP 响应，尤其适合静态文件：

- `/index.html`
- `/logo.png`
- `/app.js`

它可以负责：

- 返回文件
- gzip
- etag
- last-modified
- 304
- header 设置

---

### 15.2 反向代理

用户访问 nginx，nginx 再把请求转给后面的应用服务：

- Node.js
- Python
- Java
- Go

常见例子：

- `/api/*` -> 应用服务
- `/static/*` -> nginx 直接处理

---

### 15.3 负载均衡

如果后面有多个应用实例：

- app1
- app2
- app3

nginx 可以把流量分摊到不同机器或进程上。

---

### 15.4 站点入口控制层

它还可以做：

- HTTPS 终止
- 域名路由
- 重定向
- header 控制
- 限流
- 请求大小限制
- 安全策略

所以 nginx 更像：

- **源站前面的统一入口和调度层**

---

## 16. CDN 的作用是什么

CDN = Content Delivery Network，内容分发网络。

它最核心的价值有两点。

### 16.1 就近访问

如果源站在东京，而用户在上海、北京、新加坡、洛杉矶，直接访问源站会有更高延迟。

CDN 在不同地区部署边缘节点，用户优先从更近的节点拿资源。

结果是：

- 延迟更低
- 首屏更快
- 跨地域体验更好

---

### 16.2 减轻源站压力

热门静态资源如果每次都回源站，会让源站带宽和并发压力很大。

CDN 会在边缘节点缓存资源，后续用户请求可以直接命中边缘缓存。

这样可以：

- 降低回源次数
- 降低源站带宽成本
- 提高抗流量能力

---

## 17. nginx 和 CDN 的区别

可以用一句话区分：

- **nginx 更偏源站入口和转发服务**
- **CDN 更偏边缘缓存和全球分发**

### nginx 关注的是

- 请求进站后怎么处理
- 是直接返回文件，还是转发到后端
- 站点规则怎么配置

### CDN 关注的是

- 用户离内容近不近
- 热门资源能不能在边缘节点直接命中
- 源站压力能不能被挡住

它们通常不是替代关系，而是分层关系。

---

## 18. 一条典型链路长什么样

最常见的结构是：

```text
浏览器
  ↓
CDN
  ↓
nginx
  ↓
Node / Java / Go / Python 应用
```

### 18.1 命中 CDN 缓存时

- 浏览器请求先到 CDN
- CDN 本地就有缓存
- CDN 直接返回
- nginx 和后端应用不参与

---

### 18.2 CDN 未命中时

- CDN 没有缓存，或需要回源验证
- CDN 去找源站
- 源站通常先到 nginx
- nginx 决定：
  - 直接返回静态文件
  - 或转发给后端应用

这一步叫：

- **回源**

---

## 19. 304 可能是谁返回的

在有 CDN + nginx 的架构里，304 可能来自不同层，取决于谁在做缓存校验。

### 情况 A：浏览器直接和 nginx 协商

如果没有 CDN，或 CDN 只是透传，那么：

- 浏览器发条件请求给 nginx
- nginx 判断资源没变
- nginx 直接返回 304

---

### 情况 B：CDN 参与缓存和回源验证

可能是：

1. 浏览器请求到 CDN
2. CDN 判断自己需要校验
3. CDN 回源到 nginx
4. nginx 返回 304 或 200
5. CDN 再把结果传给浏览器

也可能是 CDN 自己就能判断并返回缓存结果。

所以单看浏览器里的一条 304，不一定 100% 能断定是哪个层亲自做出的最终判断，要结合响应头、CDN 特征头、回源策略一起看。

---

## 20. 这个场景可以怎么理解

如果响应头里出现这类信息：

- `Server: nginx/1.28.3`
- `X-Delivery-Mode: nginx-style-static-server`
- `X-Serve-Reason: nginx-style cache policy`
- `X-Region: ap-northeast-1`

那么很像这样一种情况：

- 资源是静态文件
- 服务侧采用 nginx 风格静态服务策略
- 支持 `ETag` / `Last-Modified`
- 浏览器再次访问时使用条件请求
- 资源没变，于是返回 `304 Not Modified`

这通常意味着：

- 缓存策略在按预期工作
- 并不是错误

---

## 21. 最实用的总结

### 关于缓存

- `no-cache` 不是“不缓存”，而是“缓存前必须重新验证”
- 浏览器会用条件请求去协商缓存是否还能复用

### 关于 ETag

- `ETag` 是资源版本标识
- 大概率由服务器基于文件内容或文件元数据生成
- nginx 风格静态文件场景里，常见是和文件修改时间、大小相关

### 关于 304

- `304 Not Modified` 不是错误
- 它表示资源没变，浏览器继续使用本地缓存

### 关于 `ETag` 和 `If-Modified-Since`

- 理论上只用 ETag 也可以
- 但两个一起发更稳、更兼容、更常见
- 实际里通常 ETag 是主判断，时间头是补充

### 关于 nginx 和 CDN

- **nginx**：源站入口、静态服务、反向代理、负载均衡、规则控制
- **CDN**：边缘缓存、就近访问、减少回源、提升抗压

---

## 22. 一句话版

这套机制的本质就是：

> 浏览器缓存资源，但不盲目使用；每次拿着版本信息去问服务器，服务器确认没变就返回 304，让浏览器继续用本地缓存。nginx 更像源站入口和静态服务层，CDN 更像分布在各地的加速缓存层。
