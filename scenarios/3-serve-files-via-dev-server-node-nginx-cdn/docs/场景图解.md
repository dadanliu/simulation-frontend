# 模拟场景图解

---

## 一、整体架构

这个 case 用 4 个 Node 进程模拟了一条完整的前端资源交付链路。

```
┌─────────────────────────────────────────────────────────────┐
│                          浏览器                              │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP 请求
                           ▼
┌─────────────────────────────────────────────────────────────┐
│           cdn-edge-server   :5314                           │
│                                                             │
│   模拟 CDN 边缘节点（CloudFront / Akamai 等）               │
│   · 按 region 维护独立的内存缓存                             │
│   · HIT → 直接返回缓存，附 X-Cache: HIT + Age               │
│   · MISS → 回源到 origin                                    │
└──────────────────────────┬──────────────────────────────────┘
                           │ 回源（仅 MISS 时）
                           ▼
┌─────────────────────────────────────────────────────────────┐
│         origin-nginx-server   :5313                         │
│                                                             │
│   模拟 Nginx 源站                                           │
│   · /static/* → 读磁盘文件，按 nginx 规则设置缓存头          │
│   · /api/*    → 轮询转发到 app-a 或 app-b                   │
│   · 支持 304 Not Modified（ETag / Last-Modified）            │
└─────────────┬─────────────────────────────────┬─────────────┘
              │ round-robin 负载均衡              │
              ▼                                  ▼
┌─────────────────────────┐      ┌─────────────────────────┐
│  app-instance-server a  │      │  app-instance-server b  │
│         :5311           │      │         :5312           │
│                         │      │                         │
│  模拟业务应用实例 a       │      │  模拟业务应用实例 b       │
│  返回 JSON: instance=a  │      │  返回 JSON: instance=b  │
└─────────────────────────┘      └─────────────────────────┘
```

## 二、场景 1：静态资源请求（nginx 缓存策略）

**路径**：浏览器 → CDN → Origin → 磁盘文件

```
浏览器
  │
  │  GET /js/app.12ab34cd.js
  ▼
cdn-edge-server :5314
  │
  │  edgeCaches.get("ap-northeast-1:GET:/js/app.12ab34cd.js")
  │
  ├── 有缓存 ──→ 直接返回
  │              X-Cache: HIT
  │              Age: 42          （已在边缘缓存了 42 秒）
  │              Cache-Control: public, max-age=31536000, immutable
  │
  └── 无缓存 ──→ 回源
                  │
                  │  GET /js/app.12ab34cd.js
                  ▼
            origin-nginx-server :5313
                  │
                  │  读 demo-dist/js/app.12ab34cd.js
                  │  生成 ETag = "mtimeHex-sizeHex"
                  │
                  │  文件名含 8 位 hash？
                  ├── 是（hashed asset）
                  │     Cache-Control: public, max-age=31536000, immutable
                  ├── 是 .html
                  │     Cache-Control: no-cache
                  └── 其他静态资源
                        Cache-Control: public, max-age=300
                  │
                  │  返回 200 + 文件内容
                  ▼
            cdn-edge-server
                  │  写入缓存 edgeCaches.set(key, ...)
                  │
                  │  返回给浏览器
                  │  X-Cache: MISS
                  │  Age: 0
                  ▼
            浏览器收到响应
```

**关键响应头对照**：

| 文件类型 | 例子 | Cache-Control | 浏览器行为 |
|---------|------|--------------|-----------|
| HTML | `index.html` | `no-cache` | 每次都问服务器是否有新版本 |
| hash 资源 | `app.12ab34cd.js` | `public, max-age=31536000, immutable` | 一年内不再发请求，直接用本地缓存 |
| 普通静态 | `logo.png` | `public, max-age=300` | 5 分钟内用本地缓存 |

## 三、场景 2：304 Not Modified（条件请求）

**触发条件**：浏览器已有缓存，重新验证时带上 ETag 或 Last-Modified

```
浏览器（已缓存 index.html，本地存有 ETag）
  │
  │  GET /
  │  If-None-Match: "19d7ccf5200-286"
  │  If-Modified-Since: Sat, 11 Apr 2026 13:50:54 GMT
  ▼
origin-nginx-server :5313
  │
  │  读文件，生成当前 ETag
  │  对比：请求的 ETag == 当前 ETag ？
  │
  ├── 相同 ──→ 文件未变
  │           返回 304 Not Modified
  │           响应体为空（不传文件内容，节省带宽）
  │
  └── 不同 ──→ 文件有更新
              返回 200 + 新文件内容 + 新 ETag
```

**为什么 HTML 设置 no-cache 而不是 no-store？**

```
no-cache  = "可以缓存，但每次用前都要向服务器确认一下（发条件请求）"
             → 如果没变，服务器返回 304，浏览器用本地缓存，0 字节传输
             → 既节省带宽，又保证用户拿到最新的 HTML

no-store  = "完全不缓存，每次都重新下载完整文件"
             → 更浪费带宽
```

## 四、场景 3：/api 反向代理 + 负载均衡

**路径**：浏览器 → Origin → App 实例（轮询）

```
浏览器
  │
  │  GET /api/whoami
  ▼
origin-nginx-server :5313
  │
  │  publicPath.startsWith('/api/') ？ → 是
  │
  │  upstreams = [5311, 5312]
  │  rrIndex = 0, 1, 2, 3, ...（每次请求 +1）
  │
  │  第 1 次请求：rrIndex=0，0 % 2 = 0 → 转发到 5311
  │  第 2 次请求：rrIndex=1，1 % 2 = 1 → 转发到 5312
  │  第 3 次请求：rrIndex=2，2 % 2 = 0 → 转发到 5311
  │  （循环往复）
  │
  ├──→ app-instance-server a :5311
  │         返回 { "instance": "a", "region": "ap-northeast-1", ... }
  │
  └──→ app-instance-server b :5312
            返回 { "instance": "b", "region": "ap-northeast-1", ... }
  │
  │  origin 把上游信息透传给浏览器：
  │  X-Upstream-Instance: a（或 b）
  │  X-Upstream-Port: 5311（或 5312）
  ▼
浏览器看到响应体里 instance 在 a / b 交替出现
```

## 五、场景 4：CDN 按 Region 独立缓存

**核心**：缓存 key = `region:method:pathname`，不同 region 各自独立

```
第 1 次请求（东京）
  浏览器 → GET /js/app.12ab34cd.js?region=ap-northeast-1
  cdn-edge-server
    key = "ap-northeast-1:GET:/js/app.12ab34cd.js"
    edgeCaches.get(key) → undefined（无缓存）
    → 回源 → 写缓存
    → X-Cache: MISS

第 2 次请求（东京，同一资源）
  浏览器 → GET /js/app.12ab34cd.js?region=ap-northeast-1
  cdn-edge-server
    key = "ap-northeast-1:GET:/js/app.12ab34cd.js"
    edgeCaches.get(key) → 有缓存！
    → 直接返回
    → X-Cache: HIT，Age: 3

第 3 次请求（新加坡，同一文件）
  浏览器 → GET /js/app.12ab34cd.js?region=ap-southeast-1
  cdn-edge-server
    key = "ap-southeast-1:GET:/js/app.12ab34cd.js"   ← key 不同！
    edgeCaches.get(key) → undefined（新 region，无缓存）
    → 回源
    → X-Cache: MISS                                   ← 再次 MISS
```

**缓存 Map 的状态变化**：

```
初始：edgeCaches = {}

第 1 次后：{
  "ap-northeast-1:GET:/js/app.12ab34cd.js" → { body, cachedAt: T1, ... }
}

第 3 次后：{
  "ap-northeast-1:GET:/js/app.12ab34cd.js" → { body, cachedAt: T1, ... }
  "ap-southeast-1:GET:/js/app.12ab34cd.js" → { body, cachedAt: T3, ... }
}
↑
两个 region 互不影响，就像真实 CDN 在东京和新加坡各有一个独立的 PoP 节点
```

## 六、对应真实 AWS 架构

| 本地模拟（Node 进程） | 对应 AWS 真实资源 |
|---------------------|-----------------|
| `app-instance-server a/b` | EC2 实例 a/b（运行业务代码） |
| `origin-nginx-server` | EC2 上的 Nginx（静态文件 + 反向代理） |
| `cdn-edge-server` | CloudFront 分发（全球边缘节点） |
| `?region=ap-northeast-1` | CloudFront PoP 节点位置（由用户 IP 自动决定） |
| 内存 `edgeCaches` Map | CloudFront 各 PoP 节点的缓存存储 |
| `X-Cache: HIT/MISS` | CloudFront 的 `x-cache: Hit from cloudfront` |
| `Age` 头 | CloudFront 的 `age` 头 |

## 七、一句话总结每个模拟层

```
app-instance-server  =  "我是第几台机器"
origin-nginx-server  =  "我是源站，你要文件我给，你要接口我转发"
cdn-edge-server      =  "我先查自己的缓存，没有再去问源站"
```
