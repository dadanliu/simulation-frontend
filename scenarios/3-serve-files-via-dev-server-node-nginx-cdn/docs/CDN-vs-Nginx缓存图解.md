# CDN vs Nginx 缓存图解

---

## 一、CDN 本质上是什么

**是的，CDN 就是一批分布在全球各地的服务器**（叫做 PoP，Point of Presence）。

```
你可以把 CDN 理解成：
在全球各个城市租了一堆服务器，
这些服务器存着你的静态资源的副本，
用户访问时，由离他最近的那台服务器直接响应。
```

```
                        ┌─────────────────────────────────────┐
                        │         你的源站（北京机房）          │
                        │         origin-server:443            │
                        │         demo-dist/ 的原始文件         │
                        └─────────────────┬───────────────────┘
                                          │
                 CDN 服务商（如 CloudFront / Akamai / 阿里云 CDN）
                 在全球部署了大量 PoP 节点
                                          │
              ┌───────────────────────────┼───────────────────────────┐
              │                           │                           │
              ▼                           ▼                           ▼
   ┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
   │  东京 PoP 节点    │       │ 新加坡 PoP 节点   │       │ 法兰克福 PoP 节点  │
   │  (ap-northeast-1)│       │ (ap-southeast-1) │       │   (eu-central-1) │
   │                  │       │                  │       │                  │
   │  缓存了你的       │       │  缓存了你的       │       │  缓存了你的       │
   │  JS / CSS / 图片  │       │  JS / CSS / 图片  │       │  JS / CSS / 图片  │
   └────────┬─────────┘       └────────┬─────────┘       └────────┬─────────┘
            │                          │                           │
            ▼                          ▼                           ▼
     日本用户访问               东南亚用户访问               欧洲用户访问
     延迟：5ms                  延迟：8ms                   延迟：6ms
                                                   （如果直连北京源站：100~300ms）
```

**CDN PoP 节点本质**：就是一台（或一组）装了缓存软件的服务器，
它能把从你源站拿来的资源暂存在本地，
下次同区域用户访问同一资源时直接返回，不再穿越海底光缆回源站。

## 二、Nginx 缓存 vs CDN 缓存 —— 根本区别

```
┌────────────────┬─────────────────────────────┬──────────────────────────────┐
│                │      Nginx 缓存              │      CDN 缓存                │
├────────────────┼─────────────────────────────┼──────────────────────────────┤
│ 缓存在哪里      │ 源站机器的本地磁盘/内存        │ 全球各地的 PoP 节点           │
│ 距用户多远      │ 一台机器，可能跨洲             │ 离用户最近的节点（通常同城）   │
│ 主要解决什么    │ 减少后端计算压力              │ 减少网络延迟 + 全球加速       │
│ 缓存对象        │ 动态接口响应 / 静态文件        │ 静态文件为主                 │
│ 典型使用方式    │ proxy_cache（缓存上游响应）    │ 边缘节点存文件副本            │
│ 用户感知        │ 服务器响应快了                │ 下载快了（RTT 短了）          │
└────────────────┴─────────────────────────────┴──────────────────────────────┘
```

## 三、场景对比：同一个 JS 文件，两种缓存的区别

### 场景设定

```
用户：东京（日本）
源站：北京（中国）
文件：app.12ab34cd.js（304KB）
RTT（东京→北京）：约 80ms
```

---

### 方案 A：只用 Nginx（无 CDN）

```
东京用户的浏览器
    │
    │  GET /js/app.12ab34cd.js
    │  （需要跨越东京 → 北京，~80ms RTT）
    │
    │  ────────────── 海底光缆 ──────────────────────▶
    │
    ▼
北京 Nginx 源站
    │
    │  检查本地磁盘有没有这个文件
    │  有 → 读文件，发送响应
    │
    │  ◀──────────── 海底光缆 ──────────────────────
    │
    ▼
东京用户浏览器收到文件

总耗时 ≈ 2 × RTT + 文件传输时间
       ≈ 2 × 80ms + 传输
       ≈ 160ms+ 才开始看到数据
```

**Nginx 在这里做的事**：
- 直接把磁盘上的文件读出来发给你
- 设置好缓存头（`Cache-Control / ETag`）
- 浏览器本地缓存命中时返回 304

Nginx 缓存的"缓存"，**是告诉浏览器"你可以缓存在本地"**，
但如果本地没缓存，文件还是从那台源站机器传过来的。

---

### 方案 B：Nginx + CDN

```
东京用户的浏览器
    │
    │  GET /js/app.12ab34cd.js
    │  DNS 解析到离用户最近的 CDN PoP（东京节点）
    │
    ▼
东京 CDN PoP 节点
    │
    │  检查本地缓存：有没有 app.12ab34cd.js ？
    │
    ├── HIT（缓存命中）────────────────────────────────────────────────────
    │   直接从本地内存/磁盘返回
    │   X-Cache: HIT，Age: 3600
    │   总耗时 ≈ 5ms（同城网络）
    │
    └── MISS（首次访问或缓存过期）──────────────────────────────────────────
        │
        │  回源到北京 Nginx（只有这一次需要跨越海底光缆）
        │  ─────────────── 海底光缆 ──────────────────────────────────▶
        │
        ▼
    北京 Nginx 源站
        │  返回文件
        │  ◀─────────────── 海底光缆 ──────────────────────────────────
        │
        ▼
    东京 CDN PoP 节点
        │  把文件存入本地缓存（edgeCaches.set(key, ...)）
        │  之后所有东京用户请求这个文件：命中本地缓存
        │
        ▼
东京用户浏览器收到文件
总耗时 ≈ 5ms（HIT 时）
```

## 四、关键区别图解：缓存的"位置"不同

```
方案 A：只有 Nginx

  东京用户 ──────────────────────────────────────── 北京 Nginx
               ~80ms 单程，每次请求都要跑这一趟
               缓存只存在于浏览器本地（Cache-Control 告知的）


方案 B：Nginx + CDN

  东京用户 ──── 5ms ────  东京 CDN PoP  ── 80ms（仅首次）──  北京 Nginx
                            ↑
                          缓存在这里！
                          离用户 5ms 的地方
                          第二次以后完全不用跨洋
```

## 五、两种缓存各自的职责边界

```
北京源站 Nginx 管的是：
┌────────────────────────────────────────────────────────┐
│  "这个文件能不能被缓存？缓存多久？"                      │
│                                                        │
│  Cache-Control: public, max-age=31536000, immutable    │
│       ↑                                                │
│  这是 Nginx 给 CDN 的"指示"：                          │
│  "这个文件你可以缓存 1 年，不需要来找我验证"             │
└────────────────────────────────────────────────────────┘

CDN PoP 节点管的是：
┌────────────────────────────────────────────────────────┐
│  "我要不要把它存在我这里？用户来问我，我能不能直接给？"  │
│                                                        │
│  按 Cache-Control 决定存多久                           │
│  超过 max-age 后清掉，下次请求重新回源                  │
│  返回 X-Cache: HIT / MISS 告知缓存状态                  │
└────────────────────────────────────────────────────────┘

浏览器管的是：
┌────────────────────────────────────────────────────────┐
│  "我要不要把它存在我本地磁盘？下次直接从本地读？"        │
│                                                        │
│  也是按 Cache-Control 决定                             │
│  immutable → 直接从磁盘读，不发请求                    │
│  no-cache → 每次都问（CDN 或 源站）有没有新版本         │
└────────────────────────────────────────────────────────┘
```

## 六、本 case 里的模拟对应关系

```
本 case 的代码模拟了什么：

  origin-nginx-server :5313
  └─ 模拟"源站的 Nginx"
     职责：告诉下游"这个资源能缓存、能缓存多久"
     响应头：Cache-Control / ETag / Last-Modified

  cdn-edge-server :5314
  └─ 模拟"东京 PoP 节点"（或任意一个 region 的边缘节点）
     职责：本地先查，没有再去问源站
     响应头：X-Cache: HIT/MISS，Age

  ?region=ap-northeast-1 / ?region=ap-southeast-1
  └─ 模拟"用户从不同地区发来的请求"
     真实场景里，CDN 根据用户 IP 自动判断最近的 PoP，
     这里用 query 参数手动切换，让你能直接观察
     "不同区域的缓存是独立的"这个行为


完整调用链（模拟版 vs 真实版）：

  模拟版：
  curl → cdn-edge-server :5314 → origin-nginx-server :5313 → 磁盘文件

  真实版：
  浏览器 → CloudFront PoP（东京） → EC2 上的 Nginx（北京/新加坡） → 磁盘文件
```

## 七、一句话总结

```
Nginx 缓存：
  解决"后端别老被打"的问题
  缓存的是"这台服务器上游的响应"
  用户还是要从这台服务器取数据

CDN 缓存：
  解决"用户离服务器太远"的问题
  把文件副本搬到离用户 5ms 远的地方
  用户直接从隔壁（PoP 节点）取，源站不用动

两者不矛盾，通常一起用：
  源站 Nginx 告诉 CDN "可以缓存多久"
  CDN 把文件存到全球各地的 PoP 节点
  用户从最近的 PoP 拿到文件
```
