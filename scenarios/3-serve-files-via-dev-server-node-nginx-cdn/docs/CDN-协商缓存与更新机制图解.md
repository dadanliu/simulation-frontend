# CDN 协商缓存与更新机制图解

---

## 一、协商缓存是 CDN 处理，还是到源站？

**答：CDN 自己处理，不到源站。**

协商缓存（304）的本质是：
> "浏览器问：我手里这份还是最新的吗？"
> 被问的那个服务器回答：是 / 不是。

浏览器面对的是 CDN，所以 CDN 来回答这个问题。

```
┌──────────────────────────────────────────────────────────────────────┐
│                       协商缓存的三种情况                              │
└──────────────────────────────────────────────────────────────────────┘

情况 A：CDN 有缓存，直接回 304（不回源）
─────────────────────────────────────────

  浏览器（有本地缓存，ETag="abc123"）
    │
    │  GET /js/app.12ab34cd.js
    │  If-None-Match: "abc123"
    ▼
  CDN PoP 节点（有这个文件的缓存）
    │
    │  本地缓存的 ETag 也是 "abc123"？
    │  是 → 文件没变
    │
    ▼
  HTTP 304 Not Modified     ← CDN 直接返回，不碰源站
  响应体：空
  （浏览器用本地缓存渲染）


情况 B：CDN 缓存过期，回源验证，源站返回 304
──────────────────────────────────────────────

  浏览器（有本地缓存）
    │
    │  GET /js/app.12ab34cd.js
    │  If-None-Match: "abc123"
    ▼
  CDN PoP 节点（缓存已超过 max-age，需要重新验证）
    │
    │  自己不确定文件是否有更新，去问源站
    │
    │  GET /js/app.12ab34cd.js
    │  If-None-Match: "abc123"         ← 转发浏览器的条件头
    ▼
  源站 Nginx
    │
    │  文件没变，ETag 还是 "abc123"
    ▼
  HTTP 304 Not Modified
    │
    ▼
  CDN PoP 节点
    │  刷新自己缓存的过期时间（重置 max-age 计时）
    ▼
  HTTP 304 Not Modified     ← 转发给浏览器
  （浏览器用本地缓存渲染）


情况 C：CDN 缓存过期，回源，源站有新文件
──────────────────────────────────────────

  浏览器（有本地缓存，ETag="abc123"）
    │
    │  GET /js/app.12ab34cd.js
    │  If-None-Match: "abc123"
    ▼
  CDN PoP 节点（缓存已过期）
    │  回源
    ▼
  源站 Nginx
    │
    │  文件已更新，新 ETag = "xyz789"
    ▼
  HTTP 200 + 新文件内容
    │
    ▼
  CDN PoP 节点
    │  用新文件替换本地缓存
    ▼
  HTTP 200 + 新文件内容     ← 发给浏览器
  ETag: "xyz789"
  （浏览器更新本地缓存）
```

**结论**：
```
CDN 有缓存且未过期  →  CDN 自己处理 304，不回源
CDN 缓存过期       →  CDN 带条件头回源，源站说没变 → 304
                       源站说有新版  → 200 + 新文件，CDN 更新自己缓存
```

## 二、CDN 只能处理静态资源吗？

**不是，但静态资源才是 CDN 的强项。**

```
┌─────────────────────────────────────────────────────────────────┐
│  资源类型       CDN 能缓存吗？   实际用不用？   原因              │
├─────────────────────────────────────────────────────────────────┤
│  JS / CSS       能，且强烈推荐   几乎必用       内容固定不变      │
│  图片 / 字体    能，且强烈推荐   几乎必用       内容固定不变      │
│  HTML           能，但要小心     有条件用       涉及部署时序问题  │
│  API 接口       能，但通常不用   极少用         内容跟用户状态有关│
│  WebSocket      不能             不能           CDN 不支持长连接  │
└─────────────────────────────────────────────────────────────────┘
```

### HTML 为什么"要小心"？

```
典型部署流程：

  第 1 步：上传新的 JS/CSS（文件名带新 hash）
  第 2 步：上传新的 HTML（里面引用了新 hash 的文件名）

  ┌──────────────────────────────────────────────────────────┐
  │  如果 HTML 也被 CDN 长期缓存，就会出现：                  │
  │                                                          │
  │  用户请求 HTML   → CDN 返回旧版（还引用旧 hash 的 JS）    │
  │  用户请求旧 JS   → 源站已经删了旧文件 → 404               │
  │                                                          │
  │  页面挂了！                                               │
  └──────────────────────────────────────────────────────────┘

所以实践上：
  HTML：Cache-Control: no-cache（CDN 可以存，但每次都要回源验证）
  JS/CSS with hash：Cache-Control: public, max-age=31536000, immutable
```

### API 接口为什么通常不缓存？

```
GET /api/user/profile  →  返回的是"当前登录用户的数据"
                          不同用户、不同时间，结果都不同

如果 CDN 缓存了这个接口：
  用户 A 登录，CDN 缓存了 A 的数据
  用户 B 登录，CDN 直接返回 A 的数据  ← 安全事故！

所以接口通常：
  Cache-Control: no-store  或  private
  CDN 看到这个头，不缓存，直接透传到源站
```

## 三、CDN 文件什么时候更新？

**有三种情况会触发 CDN 更新缓存：**

```
┌──────────────────────────────────────────────────────────────────────┐
│  情况 1：自然过期（max-age 到了）                                     │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  源站返回：Cache-Control: public, max-age=300                        │
│                                                                      │
│  CDN 缓存这个文件 5 分钟                                             │
│                                                                      │
│  T+0    用户访问 → CDN MISS → 回源 → 缓存                            │
│  T+100s 用户访问 → CDN HIT（Age: 100）                               │
│  T+300s 缓存到期 → 下次访问时 CDN 回源重新拿                          │
│                                                                      │
│  ┌────────────────────────────────────────────────┐                 │
│  │ 0s ───────────────── 300s ──────────────────── │                 │
│  │ │缓存有效（HIT）      │过期，回源重新验证        │                 │
│  └────────────────────────────────────────────────┘                 │
└──────────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────────┐
│  情况 2：主动 Invalidation（最常见于部署新版本后）                     │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  你在 CloudFront 控制台或 AWS CLI 发起：                              │
│  aws cloudfront create-invalidation --paths "/*"                    │
│                                                                      │
│  效果：                                                              │
│  CDN 把所有 PoP 节点上的缓存标记为"无效"                             │
│  下次任何用户请求时，重新回源                                         │
│                                                                      │
│  部署前            部署后 + Invalidation                             │
│  ────────────────────────────────────────────────────────────────── │
│                                                                      │
│  CDN 缓存：        CDN 缓存：                                        │
│  旧版 index.html   旧版 index.html（标记为 EXPIRED）                  │
│  旧版 app.abc.js   旧版 app.abc.js（标记为 EXPIRED）                  │
│                                                                      │
│  第一个用户访问 → 回源 → 拿到新版 → CDN 缓存新版                     │
│  后续用户访问 → CDN HIT（新版）                                      │
│                                                                      │
│  注意：带 hash 的文件（app.12ab34cd.js）通常不需要 Invalidation，     │
│  因为文件名变了，CDN 里根本没有新文件的缓存条目，自然回源。           │
└──────────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────────┐
│  情况 3：文件名变了（hash 变了）                                       │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  旧版：app.84d7a5c1.css                                              │
│  新版：app.f3a92bc0.css  ← 文件名不同！                              │
│                                                                      │
│  CDN 按 URL 路径作为 key 存缓存：                                    │
│  旧版的 key = "ap-northeast-1:GET:/css/app.84d7a5c1.css"            │
│  新版的 key = "ap-northeast-1:GET:/css/app.f3a92bc0.css"  ← 新 key！│
│                                                                      │
│  CDN 里根本没有新版文件的缓存                                         │
│  第一次请求新文件 → 自然 MISS → 回源拿新文件 → 缓存                  │
│  不需要手动 Invalidation！                                           │
└──────────────────────────────────────────────────────────────────────┘
```

## 四、三种情况总结图

```
Q：CDN 里的文件什么时候更新？

  文件名带 hash（JS/CSS）            文件名固定（HTML / 图片）
         │                                    │
         ▼                                    ▼
  新版文件名变了                    文件名不变，内容变了
  CDN 根本没这个 key                CDN 还存着旧内容
  自然 MISS，自动更新               需要主动处理
         │                                    │
         ▼                           ┌────────┴────────┐
  ✅ 无需任何操作                    │                 │
                              等自然过期          主动 Invalidation
                          （max-age 到期）      （部署完立刻生效）
                                    │                 │
                                    ▼                 ▼
                              下次访问时回源      全球 PoP 立刻失效
                              可能有一段时间      下一个用户访问时
                              部分用户看旧版      重新回源拿新版


前端工程化的标准做法（两者结合）：

  HTML：        no-cache + 部署后 Invalidation
                ↑ 最多一次回源就能拿到新 HTML

  JS/CSS/图片：  hash 文件名 + max-age=31536000, immutable
                ↑ 永不需要 Invalidation，自然更新
```

## 五、对应本 case 的实现

```
cdn-edge-server.js 里的更新逻辑：

  // 缓存 key 包含 pathname（不含 query）
  function cacheKey(req, region) {
    return `${region}:${req.method}:${url.pathname}`;
  }

  // 只缓存 GET 200
  if (req.method === 'GET' && result.statusCode === 200) {
    edgeCaches.set(key, { body, cachedAt: Date.now(), ... });
  }


模拟里没有实现"过期"（没有 TTL 检查），
所以一旦缓存了就一直 HIT。

真实 CloudFront 的行为：
  · 按 Cache-Control: max-age 决定缓存多久
  · 超时后下次访问重新回源
  · 也支持 API 主动 Invalidation
```
