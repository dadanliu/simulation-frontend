# recvBuf 图解 —— TCP 流重组

## 问题：TCP 不知道"帧"是什么

`socket.on('data', chunk => ...)` 每次给你的字节数**完全随机**，和帧边界毫无关系。

```
服务器发出的逻辑帧（帧头 9B + payload）：

  ┌──── 帧① HEADERS stream=1 ────┐
  │ 9B头 │←────── 77B payload ──→│   共 86B
  └───────────────────────────────┘

  ┌──── 帧② HEADERS stream=3 ────┐
  │ 9B头 │←────── 76B payload ──→│   共 85B
  └───────────────────────────────┘

  ┌──── 帧③ DATA stream=1 ───────┐
  │ 9B头 │←───── 120B payload ──→│   共 129B
  └───────────────────────────────┘
  ...
```

```
OS / TCP 实际交付给 socket.on('data') 的 chunk（任意大小）：

  chunk A = 86B        ← 刚好是帧①，运气好
  chunk B = 1643B      ← 帧②③④⑤...⑯全塞一起来了
```

**但在网络慢、包小的情况下，同一帧可能被切成很多片：**

```
  chunk #1  = 前 7B  ← 帧头都没读完（帧头 9B）
  chunk #2  = 中 7B  ← 加上前面共 14B，帧头读完了，但 payload 还差 72B
  chunk #3  = 后 7B  ← 仍然不够
  ...（要攒够 86B 才能解出帧①）
```

---

## recvBuf 的作用：攒够再解析

```
每次 data 事件：

  recvBuf = recvBuf + 新到的 chunk
               ↓
  decodeFrames(recvBuf)
  ├─ 读前 9 字节帧头 → 知道 payload 长度 L
  ├─ 如果 recvBuf.length >= 9 + L  → 取出完整帧，继续看下一帧
  └─ 如果 recvBuf.length  < 9 + L  → 字节不够，停止，把剩余字节留到下次
               ↓
  recvBuf = remaining（不完整的尾巴）
```

## 一句话总结

> **recvBuf 是一个"攒够再解析"的缓冲区。**
> TCP 给多少就攒多少，只有凑齐了一个完整帧（9B帧头 + payload），才从中取出来交给上层；
> 多余的字节留在 recvBuf，等下一个 chunk 来继续拼。
>
> 没有 recvBuf，帧头被切成两半时就会解析出垃圾数据。

## 对应代码

```javascript
socket.on('data', (chunk) => {
  recvBuf = Buffer.concat([recvBuf, chunk]);     // ← 追加，不管边界

  const { frames, remaining } = decodeFrames(recvBuf);
  recvBuf = remaining;                           // ← 不完整的尾巴留着

  for (const frame of frames) {                 // ← 只处理完整的帧
    // ...
  }
});
```

```javascript
// decodeFrames 里的停止条件：
if (offset + 9 + length > buffer.length) break; // 字节不够，等下次
```
