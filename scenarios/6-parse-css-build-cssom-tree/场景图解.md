# 场景图解 ── CSS → CSSOM 的"动画级"拆解

README 给出了骨架，这里把「规则栈是怎么一步步把 CSS 串成 CSSOM 的」
用图解 + 动画级说明再打开一层。

---

## 一、整条链路俯视：两步流水线

```
    ┌────────────────────┐     ┌─────────────────────────┐     ┌────────────────────┐
    │   CSS 字符串        │     │   Token 流              │     │   CSSOM 树         │
    │                    │     │                         │     │                    │
    │  body { margin:0 } │ ──► │  ident(body)            │ ──► │  CSSStyleSheet     │
    │  button.primary {  │     │  '{'                    │     │   ├─ StyleRule     │
    │    color: red;     │     │  ident(margin) ':'      │     │   │  └─ margin:0   │
    │    ...             │     │  number(0) ';' '}'      │     │   └─ StyleRule     │
    │  }                 │     │  ident(button) delim(.) │     │      └─ color:red  │
    └────────────────────┘     │  ident(primary) '{' …   │     └────────────────────┘
                               └─────────────────────────┘
         ── ①Tokenizer ──            ── ②CSSOM Builder ──
        （有限状态机切词）             （规则栈 push/pop 串树）
```

两步独立、可分别讲：

- 第一步：**把"无形的字符串"变成"有类型的 token"**
- 第二步：**把"扁平的 token 流"按层级关系串成一棵树**

---

## 二、Tokenizer：有限状态机

和 HTML 的 tokenizer 形态一致，但分支更少。核心状态：

```
  ┌──────────────────────────────────────────────────────────┐
  │                         DATA                              │
  │                                                           │
  │   空白 ── 连续跳过，emit ws                                │
  │   /*   ──► COMMENT                                        │
  │   " /' ──► STRING_DQ / STRING_SQ                          │
  │   @id  ── 向前读 ident，emit atKeyword                     │
  │   #id  ── 向前读 ident，emit hash                          │
  │   数字  ── 读数字 + 可选单位，emit number                   │
  │   字母  ── 读 ident；若紧跟 '(' emit function，否则 ident   │
  │   { } ( ) [ ] , ; :  ── 原样 emit 标点                     │
  │   其他  ── emit delim                                      │
  └──────────────────────────────────────────────────────────┘
       ↑                                        ↓
       └── STRING / COMMENT 读到结束符就回来 ←────┘
```

### 逐字符动作示例：`body { margin: 0; }`

| 字符 | 状态 | 动作 |
|---|---|---|
| `b` | DATA | ident 起点 |
| `o` `d` `y` | DATA | 继续扩 ident |
| ` ` | DATA | 遇到空白，emit `ident(body)`，emit `ws` |
| `{` | DATA | 原样 emit 标点 `{` |
| ` ` | DATA | ws |
| `m` | DATA | ident 起点 |
| `a` … `n` | DATA | 继续扩 ident |
| ` ` | DATA | emit `ident(margin)`，ws |
| `:` | DATA | 原样 emit 标点 `:` |
| ` ` | DATA | ws |
| `0` | DATA | number 起点 |
| `;` | DATA | emit `number(0, unit="")`，emit `;` |
| ` ` | DATA | ws |
| `}` | DATA | 原样 emit `}` |

Tokenizer 本身**不关心语法**，只关心当前字符归哪一类。

### 几个容易忽略的细节

1. **`ident(` 就是 function**
   `url(` / `var(` / `calc(` 都是在 tokenizer 这一层直接打成 `function` token，
   解析器后面不用再"回头判断是不是函数调用"。

2. **number + 紧贴的 ident/`%` 合成单位**
   `24px` 是**一个**单 token（`number(24, unit="px")`），不是三个。
   这样语法层才能直接把 value 里的长度 / 百分比当"值原子"处理。

3. **`-` 的两义性**
   - `-5px` / `-.5em` → 数字的符号
   - `-webkit-` / `--brand` → ident 的一部分
   只要看 `-` 后面第一个字符是字母就当 ident，是数字才当负号，本实现用 `input[i+1]` 一眼看穿。

4. **`#fff` 是 hash，不是 delim+ident**
   `#` 后紧跟 ident 字符 → emit `hash`；否则才 emit delim。`{` 里的 id 选择器 `#root` 走同一条路径。

## 三、CSSOM Builder：规则栈 + 语法上下文

和 HTML 的"开元素栈"类似，但这里的栈里放的是**规则容器**而不是 DOM 节点：

```
  ┌─────────────────────────────────────┐
  │             规则栈                  │
  │                                     │
  │  TOP  → StyleRule(.page-shell > main)│ ← 当前正在往里写 declarations
  │         MediaRule(@media (max-w)…)  │
  │         CSSStyleSheet                │ ← 栈底永远是 stylesheet
  └─────────────────────────────────────┘
```

但只有栈还不够。和 HTML 不一样的是，CSS 每个位置可能"期待的下一个东西"是完全不同的：
selector 还是 property？value 还是 `{`？所以还要加一个"当前语法上下文"：

```
  ┌────────────────┐    atKeyword       ┌────────────────┐
  │   TOP_LEVEL    │ ─────────────────► │   IN_PRELUDE   │
  │ / IN_MEDIA_BODY│   （开始 at-rule）   │  (收集 selector│
  │                │   其他 → selector   │   / @prelude)  │
  └────────────────┘ ◄─────────────────  └────────┬───────┘
        ▲                                         │ '{'  → 开块
        │  '}'                                    │
        │                                         ▼
  ┌─────┴──────────┐    '}' 关块          ┌────────────────┐
  │ IN_DECL_BLOCK  │ ──────────────────►  │ 回到外层 context│
  │ (读 property : │                      │  (TOP_LEVEL    │
  │   value ;)     │                      │   或 IN_MEDIA) │
  └────────────────┘                      └────────────────┘
```

所以本实现同时维护两个东西：
1. **规则栈** `stack[]`：决定"新造的子规则挂到谁下面"
2. **语法上下文** `ctx`：决定"下一个 token 怎么解释"

### 完整例子：一段 `@media` 的栈演化

输入：

```css
body { margin: 0 }
@media (max-width: 600px) {
  button.primary { width: 100% }
}
```

| 读到的 token | ctx（前 → 后） | 栈（之后） | 动作 |
|---|---|---|---|
| `ident(body)` | TOP_LEVEL → IN_PRELUDE | sheet | 开始收集 selector |
| `{` | IN_PRELUDE → IN_DECL_BLOCK | sheet > body{ | 创建 StyleRule(body)，push |
| `ident(margin)` | IN_DECL_BLOCK | sheet > body{ | 暂存 property=margin |
| `:` | IN_DECL_BLOCK | sheet > body{ | 转去读 value |
| `number(0)` | IN_DECL_BLOCK | sheet > body{ | 扩 value |
| `}` | IN_DECL_BLOCK → TOP_LEVEL | sheet | 提交 `margin:0` 后弹栈 |
| `atKeyword(@media)` | TOP_LEVEL → IN_PRELUDE | sheet | 开始 at-rule |
| `(`/`max-width`/`:`/`600px`/`)` | IN_PRELUDE | sheet | 继续收集 prelude |
| `{` | IN_PRELUDE → IN_MEDIA_BODY | sheet > @media{ | 创建 MediaRule，push |
| `ident(button)`/`delim(.)`/`ident(primary)` | IN_MEDIA_BODY → IN_PRELUDE | sheet > @media{ | 嵌套的 selector |
| `{` | IN_PRELUDE → IN_DECL_BLOCK | sheet > @media{ > button.primary{ | 嵌套 StyleRule push |
| `ident(width)`/`:`/`number(100%)` | IN_DECL_BLOCK | …同上 | 暂存 property / 读 value |
| `}` | IN_DECL_BLOCK → IN_MEDIA_BODY | sheet > @media{ | 提交 `width:100%`，弹 StyleRule |
| `}` | IN_MEDIA_BODY → TOP_LEVEL | sheet | 弹 MediaRule |

**关键点**：
- `@media` 压栈是因为它**下面还能嵌套 rule**
- `@import` 不压栈，因为它是 statement at-rule，一句话完
- 普通 StyleRule 一定会压栈，因为进到声明块后还要反复追加 declaration
- 弹栈有且只有一个触发条件：读到当前 block 的 `}`

## 四、`!important` 的处理

在 tokenizer 里，`!` 只是 `delim`，`important` 只是 `ident`，它们**不是特殊 token**。

所以本实现把 "是否 important" 的判断推迟到**拿到 value 字符串那一刻**：

```
value 原始文本： "pointer !important"
                              ^^^^^^^^^^^^^
正则剥离：       /\s*!\s*important\s*$/i
结果：           value = "pointer"，important = true
```

这和 CSS Syntax L3 规范的做法一致（规范里叫"Consume a declaration"的最后一步）。
好处：value 内部如果出现 `color: var(--x, !default)` 之类的假阳性，
只要 `!important` 不在最末尾，就不会被误判。

## 五、为什么先 tokenize、再 build CSSOM？

同样是"流水线"的理由，和 HTML parser 一致：

- **解耦复杂度**：tokenizer 不用懂语法，parser 不用处理转义 / 字符边界
- **便于错误恢复**：解析失败时"吃到下一个 `;` / `}`"就能重新同步，不至于一句写错整份崩
- **一份 token 多种消费**：tokenizer 的输出可以同时给"编辑器高亮 / 压缩器 / linter / AST parser"用
- **真实浏览器里是流式的**：浏览器读到 `<style>` 的前半段就能立刻 tokenize，
  整个 `</style>` 还没读到，前半段的 `@import` 就已经开始发起子请求了

## 六、和 HTML parser 的对应关系

| 角色 | HTML parser（场景 5） | CSS parser（本场景） |
|---|---|---|
| 状态机读字符 | `DATA / TAG_OPEN / TAG_NAME / ATTR_VALUE_DQ …` | `DATA / STRING_DQ / STRING_SQ / COMMENT` |
| 产出的 token | `StartTag / EndTag / Text / Comment / Doctype` | `ident / atKeyword / hash / number / string / function / 标点 / delim` |
| 串树的栈 | 开元素栈（存 Element 节点） | 规则栈（存 StyleRule / MediaRule 容器） |
| 压栈触发 | `StartTag` 且不是 void | 读到 `{`（开启声明块 or 开启嵌套规则块） |
| 出栈触发 | `EndTag`（向下找同名） | 读到 `}`（关闭当前块） |
| 叶子节点 | Text / Comment | Declaration（`{property, value, important}`） |
| 需要"语法上下文"吗？ | 基本不用（除 insertion mode 这些复杂分支外） | **必须**，因为同样是"ident"，在 selector 位 ≠ property 位 ≠ value 位 |

所以场景 5 和场景 6 可以理解为**同一套"流水线 + 栈"思路在两种语言上的两种实例化**。

## 七、和下一跳的接口

下一跳是 **Style 计算 / Render Tree 构建**：

```
  DOM 树 (场景 5)  ┐
                   ├──► 样式匹配 ──► 每个 DOM 节点的 computed style
  CSSOM 树 (本场景) ┘
                   └──► @import 触发的子 stylesheet 也递归参与
```

所以本场景的输出规范其实是**"给样式匹配器的一份可查询的规则列表"**：
- 每个 `StyleRule` 自带 selector（字符串，不展开）
- 每个 `Declaration` 自带 property / value / important
- `MediaRule` 是"条件容器"，样式匹配器遇到它要先判条件，再决定里面的规则是否生效

下一场景会把 DOM 和 CSSOM 合起来，做"选择器匹配 + 级联 + 继承 + 特异度裁决"，
把每个 DOM 节点上最终生效的属性集算出来。

---

## 八、一句话总结

> CSS 的解析器 = **状态机**（把字符串变成带类型的 token）+ **规则栈 + 语法上下文**（把 token 按"at-rule / style rule / declaration"的嵌套关系串成 CSSOM 树）。
>
> 手写实现把这两条主干裸露出来；postcss / 真实浏览器在同一条主干上补位置信息、保格式、错误恢复、插件管线。
