1. 用js写
2. 足够简洁
3. 每个dir有自己的package.json 用pnpm 包管理
4. 语意化的dir命名
5. 代码关键位置加注释说明逻辑
6. 参考当前dir内的 README.md 是当前场景的说明,照着README描述的内容搭建场景

# 场景搭建方法论（从 `0-read-source-and-analyze-dependencies` 提炼）

后续每个 scenario，都尽量按下面这套思路搭，不要重新长成大而全的项目骨架。

## 一、先定清楚“这一跳”到底是什么

每个场景只模拟**线性链路里的一个明确节点**。

先用一句话写清楚：

- 这一跳在整条链路里的位置是什么
- 它的输入是什么
- 它的输出是什么
- 它不负责什么

例子：

- `0-read-source-and-analyze-dependencies`
  - 负责：读取源码、分析依赖、建立模块图
  - 不负责：编译、打包、产物输出

原则：

> 一个场景只讲一个核心动作，不要把相邻几跳混在一起。

---

## 二、先写 README，再写代码

推荐顺序：

1. 先写 README
2. 在 README 里讲清楚“为什么要模拟这一跳”
3. 画图说明输入、处理过程、输出
4. 再按 README 去搭代码

README 至少应包含：

- 目标
- 目录结构
- 在线性链路中的位置
- Mermaid 图
- 怎么运行
- 和真实流行方案的对比
- 一句结论

原则：

> README 不是补充材料，而是场景设计本身。

---

## 三、代码只保留“讲清原理所必需”的最小部分

优先保留：

- 最小输入样例
- 最小核心脚本
- 最小运行命令
- 最小输出结果

优先删除：

- 和当前节点无关的 UI 层
- 不必要的框架壳子
- 多余抽象层
- 为了“像真实项目”而增加的噪音目录

用一句话判断：

> 没有它，原理讲不清吗？如果不会，就删掉。

---

## 四、目录结构要语义化，而且扁平

推荐像这样：

```text
scenario-x/
├─ README.md
├─ package.json
├─ core-script.js
└─ demo-src/
```

不要轻易长成：

```text
app/
components/
lib/
scripts/
utils/
configs/
...
```

除非当前节点真的需要它们。

原则：

> 小场景优先扁平目录，名字直接表达职责。

---

## 五、一定要有“最小输入样例”

每个场景都应该有一份能被直接读取/处理的最小输入。

例如：

- 依赖分析场景 → `demo-src/`
- 编译场景 → `input.js` / `input.jsx`
- 资源处理场景 → `assets/`
- 浏览器解析场景 → `input.html` / `input.css`

原则：

> 场景不是空讲概念，必须有可观察的输入载体。

---

## 六、核心脚本要“手写、直接、可运行”

如果目标是解释某个机制，优先自己手写一个最小实现。

例如：

- 依赖分析 → 自己写 mini-bundler
- 编译转换 → 自己写 mini-transform
- HTML 解析 → 自己写最小 tokenizer / parser

不是为了替代成熟工具，而是为了：

- 把原理裸露出来
- 让输入 -> 处理 -> 输出的路径足够短
- 便于逐行解释

原则：

> 教学场景里，最小可运行实现比“直接调用黑盒工具”更有解释力。

---

## 七、一定要和流行方案对比

每个场景都尽量包含：

- 一个手写最小实现
- 一个流行方案（Webpack / Vite / Babel / 浏览器行为等）对照

对比时重点讲：

1. 两者共同的核心动作是什么
2. 手写版故意省掉了什么
3. 流行方案多做了哪些工程能力

原则：

> 手写版负责讲透原理，流行方案负责说明真实工程里这件事是如何放大的。

推荐直接在 README 里单独加一节：

### 手写方案 vs 实际流行方案

可以固定按下面这个表来写：

| 对比项 | 手写最小方案 | 实际流行方案 |
|---|---|---|
| 目标 | 只把这一跳的核心动作讲清楚 | 在真实项目里稳定、高效、可扩展地完成这一跳 |
| 输入 | 人工收敛过的最小输入样例 | 真实项目里的多类型、多规模输入 |
| 核心动作 | 只保留主干路径 | 主干路径 + 大量边界处理 |
| 省略内容 | 错误处理、缓存、插件体系、增量更新、兼容性等 | 通常都需要补齐 |
| 输出 | 便于观察原理的最小结果 | 能直接服务真实开发和线上环境的完整结果 |
| 价值 | 便于学习、调试、逐行解释 | 便于协作、提效、稳定运行 |

对比时尽量写成下面这四句：

1. 手写方案复现了哪一个最核心动作？
2. 它为了讲清原理，故意省掉了哪些工程能力？
3. 真实流行方案在工程上额外补了哪些能力？
4. 两者的边界分别适合什么使用场景？

例如：

- `0-read-source-and-analyze-dependencies`
  - 手写方案：自己递归读文件、解析 import、产出模块依赖图。
  - 流行方案：Webpack / Vite 不只分析依赖，还会处理 alias、插件、虚拟模块、缓存、热更新和更复杂的模块解析规则。
- `1-compile-framework-syntax-to-browser-javascript`
  - 手写方案：只演示 JSX / TSX / Vue 模板如何变成 JavaScript。
  - 流行方案：Babel / SWC / Vue Compiler 还会处理 source map、语法兼容、开发调试信息、性能优化和更完整的语法覆盖。
- `2-process-css-images-and-font-assets`
  - 手写方案：只展示资源收集、重命名、路径改写、产出 manifest。
  - 流行方案：真实 bundler 还会处理 code splitting、压缩、hash 策略、postcss、loader/plugin 链路、缓存失效和 CDN 发布配合。

一句话记忆：

> 手写方案回答“本质上发生了什么”，流行方案回答“真实项目里怎样把它做得可用、稳定、可扩展”。

---

## 八、代码关键位置必须有注释，但只注释关键点

注释重点放在：

- 核心判断
- 输入输出边界
- 关键数据结构
- 为什么递归 / 为什么分支
- 这一步在线性链路里对应什么职责

不要把显而易见的语句全注释一遍。

原则：

> 注释是为了说明“为什么这样设计”，不是为了把代码逐字翻译成中文。

---

## 九、输出要能直接对应 README 里的图

最理想的状态是：

- README 里有图
- 脚本输出里的字段能映射到图
- 用户能把“源码 / 处理过程 / 输出”三者对起来看

例如依赖分析场景：

- README 画依赖图
- 脚本输出 `nodes` / `edges`
- 用户能看出来图里的每条线对应哪条 edge

原则：

> 文档图不是装饰，应该能和实际输出一一对应。

---

## 九点五、提供"可 debug 的逐步模拟"入口（推荐）

如果当前节点存在一个**内部循环过程**（比如解析、打包、依赖分析、渲染管线等），
除了"一次性跑完的 `mini` 脚本"，最好再单独提供一个 `debug/step-by-step.js` 之类的逐步脚本：

- 每一步都打印：**输入 token / 当前动作 / 关键数据结构（栈、指针、缓存）的前后状态**
- 预埋 `debugger;` 语句，支持 `node --inspect-brk` + Chrome DevTools / VSCode 单步
- 脚本也接受同一份最小输入样例（`demo-input/…`）

好处：

- 一次性脚本回答"最终结果是什么"，逐步脚本回答"过程里发生了什么"
- 读者能在真实断点里观察变量，而不是光看字符串输出
- 和 README 里的"动画级图解"（栈的 push/pop、状态机转移）能一一对应

推荐脚本形态：

```text
scenario-x/
├─ mini-xxx.js                # 一次性跑完，讲"结果"
└─ debug/
   └─ step-by-step.js         # 逐步慢放，讲"过程"
```

`package.json` 里对应：

```json
{
  "scripts": {
    "mini": "node mini-xxx.js demo-input/…",
    "debug": "node debug/step-by-step.js demo-input/…",
    "debug:inspect": "node --inspect-brk debug/step-by-step.js demo-input/…"
  }
}
```

原则：

> 如果这一跳是"内部有多步的算法"，就不仅要能看到结果，还要能被单步 debug。

---

## 十、命令要短，运行路径要单一

每个场景最好只有一个最核心命令。

例如：

```bash
pnpm mini
```

或者：

```bash
node mini-bundler.js demo-src demo-src/src/main.js
```

不要要求用户跑一串复杂步骤才能理解场景。

原则：

> 场景运行命令越短，学习路径越清晰。

---

## 十一、每个场景都要能回答这四个问题

搭完后，检查自己能不能明确回答：

1. 这一跳为什么存在？
2. 它的输入是什么？
3. 它的输出是什么？
4. 它和前后两跳的边界是什么？

如果回答不清，说明场景还不够聚焦。

---

## 十二、推荐模板

后续新场景，优先套这个模板：

```text
scenario-x/
├─ README.md              # 先写，讲目标/位置/图/运行方式/对比
├─ 场景图解.md             # 关键的"动画级"图解（可选，但强烈推荐）
├─ package.json           # 当前场景自己的依赖和命令
├─ mini-xxx.js            # 手写最小实现（一次性跑完）
├─ compare-with-xxx.js    # 与主流方案并排对比（可选）
├─ debug/
│  └─ step-by-step.js     # 逐步模拟，支持 --inspect-brk 断点（若适用）
└─ demo-input/            # 最小输入样例
```

README 结构推荐：

1. 目标
2. 在线性链路中的位置
3. 目录结构
4. Mermaid 图
5. 手写最小实现做了什么
6. 可 debug 的真实场景模拟（如适用）
7. 与流行方案对比
8. 怎么运行
9. 能回答的四个问题
10. 一句结论

---

## 最后一句总原则

> 每个 scenario 都应该像一个“最小教学实验”：
> 输入足够小，核心逻辑足够裸露，文档和输出能互相对照，既能讲清原理，又能映射到真实工程方案。
