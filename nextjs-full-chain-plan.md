# Next.js 全链路改造方案

## 1. 这次改造的核心判断

当前项目的问题不是“没有示例”，而是**示例的组织骨架错了**。

旧方案更像按术语分桶：

- RSC
- SSR
- SSG
- Build Artifacts
- Hydration
- CSR
- API Roundtrip

这种组织方式适合做知识点索引，但不适合回答这次真正要回答的问题：

> 人在 VSCode 里写下前端源码后，这些源码到底怎样一步步变成用户屏幕上的像素；用户点击后，系统又怎样一步步响应。

所以这次改造的唯一目标是：

**把项目从“按术语分类”改成“按发生顺序组织”。**

---

## 2. 新方案的唯一主线

整个项目以后围绕下面这条唯一主链展开：

`源码 -> 构建 -> 产物 -> 交付 -> 浏览器解析 -> 运行时启动 -> 首屏渲染 -> hydration -> 用户交互 -> 状态更新 -> DOM/样式/布局/绘制变化 -> 屏幕变化`

更细一点：

1. 人在编辑器里写源码
2. 源码描述状态、结构、样式、交互
3. 构建工具读取源码并分析依赖
4. TS / JSX / 模板语法被编译
5. CSS / 图片 / 字体被整理、转换、切分
6. 产出 HTML / JS / CSS / 静态资源 / 服务端逻辑
7. 这些产物由 dev server / Node / Nginx / CDN 暴露给浏览器
8. 浏览器发起请求并收到字节流
9. 浏览器解析 HTML 生成 DOM
10. 浏览器解析 CSS 生成 CSSOM
11. 浏览器执行 JS，框架运行时启动
12. 框架根据状态 / 组件树 / 路由 / 数据计算当前 UI
13. 浏览器完成 Render Tree / Style / Layout / Paint / Composite
14. 用户看到首屏
15. hydration 完成，页面进入可交互状态
16. 用户点击 / 输入 / 滚动
17. 浏览器派发事件，JS 处理函数运行
18. 应用状态变化
19. 框架重新计算受影响 UI 并更新 DOM
20. 浏览器重新进行受影响部分的渲染流水线
21. 用户看到响应结果

---

## 3. 几个关键认知

## 3.1 源码不是页面，源码是页面生成规则

源码里写的是：

- 结构规则
- 状态规则
- 样式规则
- 交互规则
- 数据读取规则

所以项目首页必须先把“源码不是像素”这件事讲清楚。

## 3.2 浏览器拿到的是字节，不是组件概念

进入网络阶段后，浏览器收到的是：

- HTML 字节
- CSS 字节
- JS 字节
- 图片 / 字体等静态资源字节
- 某些协议数据或接口响应

不能把“浏览器直接收到组件树”当成事实叙述。

## 3.3 看到页面，不等于页面已经可交互

页面能显示，主要靠 HTML/CSS 被解析与绘制。

页面能点击，还要等：

- 客户端 JS 下载
- JS 执行
- 框架运行时启动
- hydration 完成
- 事件绑定完成

所以 hydration 必须放在线性链路中间，而不是做成孤立术语卡片。

## 3.4 用户交互触发的是局部更新链

用户点击后，不是全站重新生成，而是：

- 浏览器派发事件
- JS 回调执行
- 状态变化
- 框架计算受影响 UI
- DOM / 样式 / 布局 / 绘制发生局部变化

这条链必须和首屏生成链连接起来，但不能混成一回事。

## 4. 新的文档组织原则

以后文档按下面四层组织：

### 第一层：唯一主链
先给一条总图，只回答“先发生什么，后发生什么”。

### 第二层：阶段块
把主链拆成连续阶段：

1. 源码描述阶段
2. 构建与产物生成阶段
3. 资源交付与请求阶段
4. 浏览器解析与首屏渲染阶段
5. 框架接管与可交互阶段
6. 用户交互与增量更新阶段

### 第三层：单点场景
每个阶段再拆成单独可运行场景。

这些场景是链路节点，不是术语展柜。

### 第四层：术语挂靠
`RSC / SSR / SSG / CSR / Hydration / Route Handler` 只作为局部机制说明：

- 某一段链路采用了什么机制
- 某个节点为什么这样工作

一句话：**术语从主线降级为挂件。**

---

## 5. 新的代码组织原则

## 5.1 `full-chain/` 只讲一条完整故事

主项目职责要收敛为：

> 同一个页面，从源码语义到构建产物，到浏览器解析、首屏、hydration、交互更新，是怎样串起来的。

首页需要按线性顺序展示：

1. 源码在描述什么
2. 构建阶段如何处理这份源码
3. 最终产出哪些资源
4. 浏览器请求到的是什么
5. 浏览器如何解析并画出首屏
6. hydration 如何让页面进入可交互状态
7. 用户点击后如何走本地更新链或请求往返链

## 5.2 `scenarios/` 改为“链路节点场景”

短期内可以保留现有目录名，但每个场景必须在 README 和页面内明确：

- 它在线性链路中的位置
- 它模拟的真实节点
- 它的输入是什么
- 它的输出是什么
- 它与前后节点的关系是什么

也就是说，目录名可以暂时保守，但叙事必须已经切到线性链路视角。

## 5.3 每个场景必须满足三个要求

1. 可以单独运行
2. 只解释一个节点或一个很短的连续片段
3. 页面上能直接看到这个节点的职责、输入、输出、观察点

## 6. 本次代码改造范围

### 6.1 文档层

先改：

- `plan.md`
- `nextjs-full-chain-plan.md`
- 根 `README.md`
- `full-chain/README.md`
- 各 `scenarios/*/README.md`

目标：先统一叙事骨架。

### 6.2 主项目层

`full-chain/` 首页改成“线性故事页”，不再只是几个术语卡片。

最少要出现：

- 源码语义区
- 构建与产物区
- 交付与字节区
- 浏览器解析区
- 首屏渲染区
- hydration / 可交互区
- 本地状态更新区
- 请求往返更新区

### 6.3 独立场景层

现有目录暂不强制重命名，但要把页面内容和 README 改成“节点视角”：

- `browser-receives-server-component-result`：服务端读取数据并参与首屏生成
- `build-emits-static-html`：构建期提前产出静态结果
- `build-emits-runtime-artifacts`：同一份源码如何被拆成不同产物
- `request-generates-html-on-server`：请求到来时重新生成首屏
- `hydration-enables-interaction`：先能看、后能点
- `client-state-drives-ui-update`：客户端状态如何驱动局部 UI 更新
- `fetch-roundtrip-updates-ui`：浏览器和服务端的一次完整请求往返

---

## 7. 验收标准

改造完成后，项目应满足三类验收：

### 7.1 看文档时
读者能顺着讲清：源码 -> 构建 -> 产物 -> 交付 -> 浏览器解析 -> 首屏 -> hydration -> 交互更新。

### 7.2 看代码时
读者能从目录、命名、README、页面说明看出：

- 这个文件属于链路哪一段
- 这个页面模拟哪个节点
- 输入和输出是什么

### 7.3 运行页面时
读者能通过真实按钮、时间戳、说明文字和请求结果观察到：

- 首屏服务端生成
- 构建或静态语义
- hydration 接管
- 本地状态更新
- 请求往返更新
- 浏览器渲染链被再次触发

---

## 8. 一句话结论

这次改造不是“多补几个知识点”，而是换掉整个项目的叙事骨架：

**从按术语分类的示例集，改成按时间顺序展开的前端全链路模拟系统。**
